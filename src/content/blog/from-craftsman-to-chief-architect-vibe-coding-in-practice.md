Vibe Coding 这个词很容易让人误解。很多人一听到它，想到的是“随便说几句，让模型自己把活干完”。如果是这样理解，它很快就会变成另一种形式的撞大运。

我越来越倾向于把 Vibe Coding 理解为一门新的工程实践：它当然保留了人与模型协作时那种流动、快速、富有直觉的工作感，但真正决定质量的，从来不是 vibe 本身，而是你有没有能力把这种流动感装进一套稳定的工程纪律里。

换句话说，Vibe Coding 不是放弃工程，而是把工程的重心从“我如何亲手写完每一行代码”，转向“我如何设计一套让模型持续做对事的系统”。

## 先澄清一个误区：Vibe Coding 不等于让模型自由发挥

如果一个任务的边界不清、验收标准不清、上下文混乱，那模型的高效率只会把混乱放大。

很多人第一次用大模型写代码，会有一种很强的快感：需求一说，代码就出来了；测试一跑，似乎也能过；页面一看，好像也能用。问题在于，这种“看起来差不多”的体验，经常会掩盖三个事实：

1. 你并没有真正定义问题。
2. 你并没有真正约束过程。
3. 你并没有真正建立验收。

所以 Vibe Coding 的第一原则不是“快”，而是 **先把不确定性装进边界里**。只有边界清楚，速度才有意义。

## 真正有用的经验一：任务必须显式化，不能只存在于聊天上下文里

我现在越来越不信任“把任务交给对话历史记住”这件事。因为对话上下文天然是流动的、脆弱的，而且很容易被后续内容覆盖或扭曲。

如果一项工作需要持续几轮以上协作，我会默认它必须拥有可见的任务契约。这个契约可以是一份 `plan.md`，也可以是一份 story 文档、任务清单、验收说明，总之它应该满足几个条件：

- 人能读。
- 模型能读。
- 当前状态清楚。
- 下一步动作清楚。
- 完成标准清楚。

这样做的意义非常大。因为一旦任务被显式写出来，协作双方就不再依赖记忆，而依赖同一份外部事实。很多所谓“模型不稳定”，其实并不是模型真的不稳定，而是人和模型从一开始就没有共享同一个清晰任务。

## 真正有用的经验二：把大任务拆成可验证的小单元

Vibe Coding 失败的一个高频原因，是把一句很大的目标直接交给模型，希望它一步到位。

这在简单脚本上偶尔有效，但在稍微复杂一点的开发任务里，几乎总会失控。因为大模型擅长局部展开，不擅长在长链路里始终保持精确。你给它一个模糊的大目标，它就会用大量合理但未必正确的细节把空白补满。

更稳妥的方式不是追求“一次生成完整答案”，而是把问题拆成一组更容易验证的中间成果，例如：

1. 先确认问题定义是否准确。
2. 再确认方案拆解是否合理。
3. 再生成某一个局部模块。
4. 再验证这个局部模块是否符合预期。
5. 最后再把局部结果拼回整体。

这种方法看起来慢一点，但整体上往往更快。因为你减少了返工，也减少了那种“看上去完成了，实际上埋了一堆错”的伪进度。

## 真正有用的经验三：测试不能只放在最后，而要贯穿生成过程

如果测试只发生在最后一步，那么在 AI 参与开发的场景里，你会经常遇到一种情况：结果坏了，但你不知道是哪里开始坏的。

所以我更认可双层验证。

第一层是整体验证。也就是从用户视角看，这个功能、这个页面、这个流程，最终是否真的成立。

第二层是局部验证。也就是在每一个关键中间步骤，检查它是否还在正确轨道上。比如数据结构有没有变形，接口语义有没有漂移，重构后行为有没有偷偷改变，提示词更新后输出格式是否还稳定。

这一层局部验证非常关键，因为 AI 最擅长制造一种“表面过关、内部偏移”的错觉。它不是明显报错，而是悄悄把一些细节改歪。没有中间检查，你通常要等到很后面才发现问题，而那时修复成本已经很高了。

## 真正有用的经验四：不同职责最好由不同上下文承担

我越来越少让同一个对话线程同时承担“设计、编码、测试、复盘、记录状态”这几种角色。不是因为做不到，而是因为这样做会让上下文迅速污染。

一个线程越想同时处理所有事情，它的状态就越容易混乱。它会开始混淆什么是已完成、什么是待验证、什么是假设、什么是事实。到后面你会发现，不是模型不会写，而是它已经搞不清自己到底在做哪一类工作。

更稳的办法，是做上下文隔离：

- 让一个上下文专注于设计和拆解。
- 让一个上下文专注于实现。
- 让一个上下文专注于验证和挑错。
- 让外部文档承担状态记录。

这件事的本质不是“多开几个窗口”，而是给不同认知任务建立边界。边界一清楚，产出的稳定性通常会明显提升。

## 真正有用的经验五：先写验收，再让模型动手

过去我们常说“先写测试，再写代码”。在 Vibe Coding 语境里，我会把它扩展成一句话：**先写验收，再让模型生成。**

这里的验收不一定总是自动化测试，它也可以是更广义的完成标准，比如：

- 这个功能必须保留哪些旧行为。
- 这个改动不能破坏哪些接口约定。
- 这个页面必须满足哪些交互结果。
- 这篇文档最后必须回答哪些问题。

如果这些东西不先定义清楚，模型就只能凭自己的统计直觉补全任务。而统计直觉最擅长的是“像”，不是“准”。

一旦你把验收写清楚，模型的角色就会发生变化。它不再是一个凭感觉乱试的生成器，而更像一个朝着明确靶心持续逼近的执行者。

## 更深的思考一：Vibe Coding 其实在重新定义“会写代码”

以前我们评价一个工程师，常常会看他是否能快速、熟练、稳定地把代码写出来。这当然仍然重要，但已经不够了。

在 AI 参与之后，“会写代码”开始被拆成两层能力：

第一层是传统能力，也就是理解系统、定位问题、写出正确实现。

第二层是新的能力，也就是把任务表达成模型能正确执行的形式，并能持续验证、纠偏、约束它。

后者其实越来越重要。因为模型已经能替你完成大量局部实现，真正稀缺的，不再是单点编码速度，而是 **把复杂目标变成可执行系统的能力**。

所以今天一个强工程师，未必是写得最多的人，而更可能是最会定义目标、拆解工作、设置约束和建立反馈回路的人。

## 更深的思考二：从控制代码，转向控制系统

这几年我最明显的感受是，工程师的控制感正在转移。

过去的控制感来自细节。你亲手写每个函数，知道每个分支为什么这样判断，所以你觉得系统在你掌控中。

现在的控制感更多来自结构。你不一定亲手写完所有代码，但你定义了边界、流程、接口、测试、状态和回滚方式，所以你依然知道整个系统如何运转。

这是一种很不一样的能力。它要求你接受一件事：你可能会对某些具体代码产生“熟悉的陌生感”。你知道它为什么存在，也知道它是否合格，但它未必是你逐字逐句敲出来的。

如果不能接受这一点，人很容易重新退回“我还是自己写吧”的旧路径。可一旦接受，你就会发现自己获得了更大的杠杆。你失去的是对代码行的绝对占有，换来的是对复杂系统更高层级的调度权。

## 更深的思考三：Vibe Coding 的“道”不是技巧，而是秩序

如果只把 Vibe Coding 当成一组 prompt 技巧，它很快就会过时。模型升级一次，界面换一次，很多表层方法就会失效。

但它背后真正稳定的东西，其实更接近“道”层面的判断：

- 先定义问题，再追求答案。
- 先建立约束，再追求速度。
- 先设计反馈，再追求自动化。
- 先把状态外化，再让多人或多 Agent 协作。
- 先承认不确定性，再用评测和验证把它缩小。

这些原则并不会因为模型换代而失效。相反，模型越强，这些原则越重要。因为能力越强的系统，一旦缺少秩序，就越容易把错误放大得更快、更远。

所以我理解的 Vibe Coding 之“道”，不是“如何更会用模型”，而是“如何在高生成力的世界里，仍然保持清醒的工程秩序”。

## 写在最后

Vibe Coding 最吸引人的地方，是它让很多原本沉重的工作突然变得轻盈了。你可以更快试错，更快实现，更快看到结果。

但它真正有价值的地方，不是让人沉迷这种轻盈感，而是逼着工程师升级自己的角色。你不能只做代码的手艺人了，你得开始做任务的定义者、流程的设计者、验收的制定者、系统的建筑师。

当代码越来越容易被生成，真正稀缺的能力就不再只是“写”，而是“判断写什么、怎么拆、如何验、出了问题如何收束”。这也是为什么我越来越觉得，Vibe Coding 最终考验的不是手速，而是结构感；不是技巧，而是方法；不是一时的灵感，而是长期稳定的秩序感。

这正是从“手艺人”走向“总建筑师”的过程。
